# 提案质量检查清单（质量门 A）

本清单供 Phase **1f（质量门 A，提案自检）** 使用。审查方对每条给出 **PASS** / **FAIL** / **N/A**（不适用须说明理由）。**任一 FAIL 且非误报时，整体验收为 FAIL。**

与 `references/quality-checklist.md`（代码质量门 B/C）同构，审查维度面向提案/设计制品。

## 检查项表

| ID | 分类 | 检查项 | 判定说明 | 适用制品 |
|----|------|--------|----------|----------|
| PA01 | 准确性 | 技术描述正确 | 技术方案无事实错误、无自相矛盾；引用的接口/模块与实际代码一致。 | proposal/design |
| PA02 | 准确性 | 术语与需求对齐 | 提案用词与用户原始需求一致，无偷换概念或语义偏移。 | proposal |
| PA03 | 完整性 | 需求功能点覆盖 | 用户原始需求中的每个功能点在提案中有对应方案，无遗漏主路径。 | proposal |
| PA04 | 完整性 | 边界场景覆盖 | 对异常流、空值、并发、权限等边界有明确设计或声明。 | design |
| PA05 | 完整性 | 验收标准齐全 | 每个 task 有明确可验证的验收标准（非模糊的「实现完成」）。 | tasks |
| PA06 | 结构清晰度 | 任务拆分合理 | 单个 task 不超过 1 小时工作量；职责边界清晰，无大杂烩 task。 | tasks |
| PA07 | 结构清晰度 | 依赖关系正确 | `depends_on` 反映真实依赖，无循环依赖，无遗漏前置。 | tasks |
| PA08 | 结构清晰度 | 文档组织合理 | proposal → design → specs → tasks 层次分明，不混杂。 | 全部 |
| PA09 | 可操作性 | task 描述可独立执行 | SubAgent 仅凭 task 描述 + session 上下文即可开始工作，无需反向猜测。 | tasks |
| PA10 | 可操作性 | 非目标明确 | 明确声明本次变更不包含的范围，防止 scope creep。 | proposal |
| PA11 | 范围限定 | 无需求蔓延 | 提案未引入用户未要求的额外功能或大范围重构。 | proposal/design |
| PA12 | 范围限定 | 无冗余制品 | 不包含无关文档、测试数据、示例文件等冗余产出。 | tasks |
| PA13 | 完整性 | 需求偏离度 | 提案是否包含需求未提及的额外功能（scope creep）。与 PA11 互补：PA11 审查设计/方案层面的功能蔓延，PA13 聚焦需求覆盖率——是否有提案功能点在原始需求中找不到对应来源。 | proposal |
| PA14 | 准确性 | 术语一致性 | 提案关键术语是否与用户原始需求用词一致，无偷换概念。与 PA02 互补：PA02 审查整体语义对齐，PA14 聚焦具体术语——关键名词/动词是否被替换为同义但易混淆的表述。 | proposal |

## 使用说明

1. **质量门 A（1f）**：对所有检查项逐条判定。
2. **N/A**：仅当该项对当前提案类型客观不适用时使用，并在 `evidence` 中写明原因。
3. **输出格式**与代码质量门一致：`{"items": [{"id": "PA01", "result": "PASS|FAIL|N/A", "evidence": "..."}], "summary": "PASS|FAIL"}`
4. 任一 FAIL → summary 为 FAIL。FAIL 时 evidence 必须指出具体问题位置。
