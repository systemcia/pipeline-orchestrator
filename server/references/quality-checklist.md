# 质量检查清单（质量门 B / C）

本清单供 Phase **3-f（质量门 B，task 级）** 与 **4b（质量门 C，全局级）** 使用。审查方对每条给出 **PASS** / **FAIL** / **N/A**（不适用须说明理由）。**任一 FAIL 且非误报时，整体验收为 FAIL。**

- **task**：针对当前 task 变更文件内的实现细节。
- **global**：针对本次编排涉及的全部变更与会话上下文（跨文件、跨 task、与需求/设计整体对齐）。
- **both**：在 task 级与全局级审查中均需判定（视角不同：局部实现 vs 整体一致）。

## 检查项表

| ID | 分类 | 检查项 | 判定说明 | 级别 |
|----|------|--------|----------|------|
| C01 | 正确性 | 逻辑正确 | 控制流与数据流与意图一致；无颠倒条件、错误默认值、漏分支等。 | both |
| C02 | 正确性 | 边界与空值 | 对 nil/空集合、零值、极值、空字符串等边界有明确行为，无静默错误。 | both |
| C03 | 正确性 | 类型与契约 | 类型使用正确；函数/方法签名与调用约定一致，无隐式弱类型滥用。 | task |
| C04 | 正确性 | 资源与生命周期 | 文件、连接、锁、句柄等获取与释放配对；无泄漏或未关闭路径。 | both |
| C05 | 正确性 | 并发安全 | 共享可变状态有合适同步或不可变设计；无数据竞争、死锁明显风险。 | both |
| S01 | 安全 | 输入校验 | 外部输入（参数、请求体、环境变量等）经校验或规范化后再使用。 | both |
| S02 | 安全 | 敏感信息 | 无密钥、token、密码硬编码或写入日志/错误信息；无意外泄露路径。 | both |
| S03 | 安全 | 权限与越权 | 鉴权/授权边界清晰；无跳过检查或信任客户端越权操作的路径。 | both |
| E01 | 错误处理 | 错误传播 | 错误被返回或包装上传，不被吞掉；调用链上语义可追踪。 | both |
| E02 | 错误处理 | 日志与可观测 | 关键失败路径有适当日志或指标钩子（级别与信息量合理）。 | both |
| E03 | 错误处理 | 降级与容错 | 依赖失败时有明确策略（重试/熔断/默认值/失败返回），而非未定义行为。 | both |
| T01 | 可测试性 | 可测结构 | 核心逻辑可单测或与 IO 边界可分离；避免不可测的硬编码全局状态。 | task |
| T02 | 可测试性 | 测试覆盖 | 对本次变更的关键路径存在对应测试或等价验证（若无测试框架则 N/A 须说明）。 | both |
| M01 | 可维护性 | 命名与可读性 | 标识符语义清晰；无误导性命名；复杂表达式有合理拆解。 | task |
| M02 | 可维护性 | 函数与文件尺度 | 单函数/单文件职责集中；无明显「上帝函数」或难以导航的巨型文件。 | task |
| M03 | 可维护性 | 模块耦合 | 依赖方向合理；无循环依赖、不当跨层调用或脆弱的全局耦合。 | global |
| M04 | 可维护性 | 重复与复用 | 无显著重复逻辑；应复用的抽象已抽取或引用一致实现。 | both |
| A01 | 需求/设计对齐 | 接口与约定一致 | 公开 API、配置项、事件契约与 design/session 描述一致，无静默破坏。 | both |
| A02 | 需求/设计对齐 | 需求覆盖 | 用户原始需求与验收要点在本次变更集中得到覆盖，无遗漏主路径。 | global |
| A03 | 需求/设计对齐 | 范围限定 | 变更未引入与 task/会话目标无关的大范围重构或功能蔓延。 | both |
| P01 | 性能基础 | 明显反模式 | 无在热路径上的明显 O(n²) 可优化、无阻塞式 sleep 轮询等低级问题。 | both |
| P02 | 性能基础 | 数据结构选择 | 对规模有基本合理的数据结构与算法选择（如集合查找、批量处理）。 | both |
| P03 | 性能基础 | 数据库查询 | 新增/修改的数据库查询无 N+1 问题、有合理索引、无全表扫描（涉及 DB 操作时）。 | both |
| P04 | 性能基础 | 并发与 I/O | 可并行的 I/O 操作未串行阻塞；批量操作优于逐条循环（涉及网络/文件 I/O 时）。 | task |

## 可选: 性能基准测试（项目配置 `benchmark_cmd` 时启用）

当 `.pipeline-orchestrator.yaml` 配置了 `benchmark_cmd` 字段时，Phase 3 后置检查在 d-3 回归测试后执行：

```
[Shell] {benchmark_cmd}
[Shell] $O test-gate $DIR benchmark '{"passed": true/false, "shell_exit_code": N, "output": "..."}'
```

FAIL 为 **SOFT_FAIL**（记 pending，不阻塞）。此步骤仅在项目显式配置时启用，不默认执行。

## 使用说明

1. **质量门 B（3-f）**：仅对级别为 **task** 或 **both** 的条目逐条判定（审查文件为 `{files}`）。
2. **质量门 C（4b）**：仅对级别为 **global** 或 **both** 的条目逐条判定（结合 `session.md` 与全部变更文件）。
3. **N/A**：仅当该项对当前语言、框架或变更类型客观不适用时使用，并在 `evidence` 中写明原因；不得用 N/A 回避明显问题。
4. 输出格式以 `references/output-schemas/quality-review.json` 为准。
